Lock SystemCheckpoint 기반 Merge Node 충돌 방지 — Deadlock Zone, Priority-Aware Lock

1. 개요

VPS Lock System은 merge node(2개 이상 incoming edge가 합류하는 지점)에서 차량 충돌을 방지하기 위한 Checkpoint 기반 Lock 메커니즘입니다.

핵심 원리

  • 차량이 merge node에 접근하면 LOCK_REQUEST 를 발행
  • 해당 node에 다른 holder가 없으면 즉시 GRANT, 있으면 WAIT
  • Holder가 merge를 통과하면 RELEASE → 다음 차량에 grant
  • Deadlock Zone은 게이트 기반 자동 진입/탈출 관리

2. Checkpoint 시스템

Checkpoint는 차량 경로 위에 자동 배치되는 제어 지점입니다. Checkpoint builder가 path 기반으로 자동 생성하며, 각 checkpoint에는 bitmask flag가 할당됩니다.

Checkpoint Flags (Bitmask)

FlagHexBit설명
LOCK_REQUEST0x01bit 0Merge node lock 요청 지점
LOCK_WAIT0x02bit 1Lock 대기 (정지) 지점
LOCK_RELEASE0x04bit 2Lock 해제 지점
MOVE_PREPARE0x08bit 3감속 준비 구간 진입
MOVE_SLOW0x10bit 4저속 통과 구간

Checkpoint Memory Layout

차량당 801 floats

[count] [cp0_edge, cp0_ratio, cp0_flags, cp0_targetEdge] [cp1_edge, ...] ... [cp199_edge, ...]

count        : 1 float  (실제 checkpoint 개수)
checkpoint   : 4 floats (edge, ratio, flags, targetEdge)
max          : 200 checkpoints
total        : 1 + (200 × 4) = 801 floats per vehicle
FieldType설명
edgefloat (1-based)해당 checkpoint가 위치한 edge index
ratiofloat (0~1)edge 위 위치 비율 (0=시작, 1=끝)
flagsfloat (bitmask)Checkpoint flag 조합 (위 표 참조)
targetEdgefloat (1-based)Lock 대상 merge edge (flags에 LOCK이 포함된 경우)

3. Lock 라이프사이클

Lock은 REQUEST → GRANT → WAIT → RELEASE 4단계 사이클로 동작합니다.

REQUEST GRANT WAIT RELEASE | | | | v v v v merge node 접근 큐가 비어있거나 다른 holder 존재 merge 통과 완료 ↓ 큐 등록 첫 번째 요청자 ↓ ↓ ↓ 즉시 holder 배정 VELOCITY = 0 grantNextInQueue() STOPPED = true ↓ LOCKED = true 다음 차량 GRANT

3.1 REQUEST

  • 차량이 merge node의 LOCK_REQUEST checkpoint를 통과하면 큐에 등록
  • 큐가 비어있으면 즉시 GRANT — holder로 지정
  • 큐에 이미 다른 차량이 있으면 WAIT 상태로 진입

3.2 WAIT

  • 현재 holder가 다른 차량이면 요청 차량은 완전 정지
  • VELOCITY = 0, STOPPED = true, LOCKED = true
  • WAIT 상태의 차량은 holder가 release할 때까지 대기

3.3 GRANT

  • Holder가 release하면 grantNextInQueue() 호출
  • 기본 정책: FIFO (선착순)
  • Deadlock Zone의 경우: 접근 차량 우선 (proximity 기반)

3.4 RELEASE

  • Holder가 merge node를 완전히 통과하면 lock 해제
  • 다음 대기 차량에 자동 grant
  • Auto-release: pendingReleases 맵에 등록된 releaseEdge 도달 시 자동 해제

4. Pre-Lock (시뮬레이션 시작)

시뮬레이션 시작 시 이미 merge node 근처에 있는 차량들의 충돌을 방지하기 위해 preLockMergeNodes()가 실행됩니다.

preLockMergeNodes() | v merge 방향 edge의 차량 수집 | v ratio 내림차순 정렬 (merge에 가장 가까운 차량이 첫 번째) | +--> 가장 가까운 차량 → holder 지정 | +--> 나머지 차량 → 큐 등록 (순서대로) | +--> 5.1m 이내 비-holder → 강제 정지 (VELOCITY=0)

주의: Pre-lock 단계에서 PRELOCK_STOP_DISTANCE (5.1m) 이내에 있는 비-holder 차량은 즉시 강제 정지됩니다. 이는 시뮬레이션 시작 직후 충돌을 원천 차단합니다.

5. Deadlock Zone 시스템

Deadlock Zone(DZ)은 일반 merge lock만으로는 교착 상태를 해결할 수 없는 복합 교차 구간을 관리합니다.

5.1 DZ 탐지 조건

1-Hop DZ

incoming edge 2개 이상 + 고각도 (>90°) 진입 → DZ merge 지정

2-Hop Curve Passthrough

짧은 LINEAR 체인 안에 포함된 DZ → waitRelocations 처리

5.2 Gate 자동 처리

updateDeadlockZoneGates() 가 매 프레임 실행되며, DZ 경계 edge를 기준으로 자동 진입/탈출을 관리합니다.

updateDeadlockZoneGates() [매 프레임] | +--> 진입 edge 감지 | | | +--> auto REQ | +--> auto GRANT (빈 zone) | +--> auto BLOCK (점유 중) | +--> 탈출 edge 감지 | +--> auto RELEASE

5.3 Holder Swap (교착 해소)

Deadlock 감지 & Swap: Holder가 2초 이상 stuck 상태이고, 큐에 ready 상태의 다른 차량이 있으면 → 강제 holder swap 실행

  • detectAndSwapDeadlockedHolders() 가 매 프레임 실행
  • Stuck 임계값: HOLDER_STUCK_THRESHOLD_MS = 2000ms
  • Swap 시 기존 holder는 큐 뒤로 이동, 새 holder가 통과

6. Priority-Aware Lock (경로 변경)

차량이 경로를 재탐색(reroute)할 때, 기존 lock을 정리하고 새 경로에 맞게 lock을 재설정하는 메커니즘입니다.

processPathChange() | +--> releaseOrphanedLocks() | | | +--> 새 경로에 없는 lock → 해제 | +--> 20m 이상 우회 lock → 해제/취소 | +--> requestLockWithPriority() | +--> 거리 기반 큐 삽입 (FIFO 위반 허용) +--> Holder swap: 1m 이상 가까우면 + ratio < 0.5 안전가드

6.1 Orphaned Lock 해제 조건

조건동작
새 경로에 해당 merge node가 없음즉시 release
새 경로 상 merge까지 거리 > MAX_DIRECT_MERGE_DIST (20m)release / cancel

6.2 Priority 큐 삽입

  • 일반 lock은 FIFO 큐 — 먼저 요청한 차량이 먼저 grant
  • 경로 변경 시 거리 기반 삽입 허용 — merge에 더 가까운 차량이 큐 앞으로
  • Holder swap 안전가드: ratio < 0.5 (holder가 merge 전반부에 있을 때만 swap 허용)
  • Swap 최소 거리: 새 차량이 기존 holder보다 1m 이상 가까워야 함 (SWAP_SAFETY_MARGIN)

7. 프레임별 처리 순서

매 시뮬레이션 프레임마다 Lock System은 아래 순서로 처리됩니다.

Frame Update Order

1. checkAutoRelease()
   → pendingReleases 맵 순회, releaseEdge 도달 차량 자동 해제

2. updateDeadlockZoneGates()
   → DZ 경계 edge 기반 자동 REQ/GRANT/BLOCK/RELEASE

3. detectAndSwapDeadlockedHolders()
   → 2초 이상 stuck holder 감지, ready 차량과 swap

4. for each vehicle: processCheckpoint()
   → 각 차량의 현재 checkpoint flags 처리
   → LOCK_REQUEST/LOCK_WAIT/LOCK_RELEASE 각각 handler 호출

10. 내부 상태 구조 (LockMgrState)

interface LockMgrState {
  // Core lock state
  locks:            Map<nodeName, holderVehId>     // 현재 holder 맵
  queues:           Map<nodeName, vehId[]>          // 대기 큐 (FIFO)

  // Auto-release tracking
  pendingReleases:  Map<vehId, { node, releaseEdge }>  // 자동 해제 대기

  // Wait state
  waitingVehicles:  Set<vehId>                      // WAIT 상태 차량 집합

  // Deadlock Zone
  deadlockZoneMerges: Map<nodeName, DZConfig>       // DZ merge 설정
  dzOccupancy:        Map<zoneName, vehId | null>    // DZ 점유 상태
  holderStuckTimers:  Map<nodeName, timestamp>       // Holder stuck 시간 추적

  // Priority / Path Change
  vehicleLockMap:   Map<vehId, Set<nodeName>>       // 차량별 보유 lock 목록
}

상태 관계도

locks (holder) <---> queues (waiting list) | | | grantNextInQueue() | enqueue on REQUEST | | v v pendingReleases waitingVehicles (auto-release edge) (VELOCITY=0, STOPPED) | | checkAutoRelease() | v RELEASE → grant next → dequeue

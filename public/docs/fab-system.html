Part 1 — FAB System

1.1 FabContext (Fab별 컨테이너)

Fab 1개 = FabContext 1개. 각 Fab은 완전히 독립된 시뮬레이션 컨테이너입니다.

관리 데이터 구조

  • VehicleDataArray — 차량 상태 (위치, 속도, 엣지, ratio 등)
  • SensorPointArray — 센서 충돌 검출용 좌표
  • EdgeVehicleQueue — 엣지별 차량 대기열

매니저

  • LockMgr — 합류점 Lock 관리
  • TransferMgr — 반송 명령 / 오더 관리
  • AutoMgr — 자동 라우팅 / 스테이션 배차
  • EdgeStatsTracker — EWMA 엣지 통계 수집

초기화 흐름

setSharedBuffer └─ SharedArrayBuffer 바인딩 (VehicleDataArray, SensorPointArray) initializeVehicles └─ 차량 초기 배치 (edge/ratio/speed 설정) buildBayLoopMap └─ Bay 루프 맵 구축 (Idle 정책용) lockMgr.init └─ merge node 탐지, Lock 테이블 초기화 preLockMergeNodes └─ 사전 Lock 가능 노드 등록 autoMgr.initStations └─ 스테이션 목록 로드 + 배차 가능 지점 설정

1.2 Simulation Step Pipeline (5단계)

매 프레임 실행되는 핵심 파이프라인:

Step 1: Collision Check └─ markStop() — SAT 기반 센서 충돌 → 정지 명령 Step 2: Lock Management └─ lockMgr.updateAll() — 합류점 진입 제어 (FIFO/BATCH) Step 3: Movement Update └─ position/edge transition + EWMA tracker 갱신 가속→감속→정지→재출발 물리 사이클 Step 4: Auto Routing └─ autoMgr.update() + lockMgr.processPathChange() Dijkstra 재탐색, 경로 변경 시 Lock 재조정 Step 5: Render Buffer Write └─ worker→render copy — Main Thread 읽기용 버퍼에 기록

1.3 SimulationEngine (Master Controller)

구조

  • fabContexts: Map<fabId, FabContext> — 모든 Fab 인스턴스 보유
  • Main loop: setInterval → step(delta) → for each fab: fab.step()
  • simulationTime 누적 (ms 단위, 시뮬레이션 경과 시간)
  • Performance reporting: 5초마다 step 통계 수집 및 리포트

1.6 Per-Fab Configuration

Global default → Fab override (partial patch) 방식으로 Fab별 독립 설정을 지원합니다.

fabConfigStore

  • lock — Lock 전략, request/wait distance
  • movement — 가감속, 최대속도, curve speed
  • sensor — 센서 존 크기, hitZone 임계값
  • routing — Dijkstra 주기, BPR 파라미터
  • transfer — 오더 배차, idle policy

Runtime update: Worker message (SET_ROUTING_CONFIG, SET_LOCK_CONFIG 등)로 즉시 반영.

Part 2 — Real-time Statistics

2.1 Order Statistics Pipeline

Worker → FabContext.flushOrderStats() every 2000ms.

OrderStatsData 구조

  • completed — 누적 완료 오더 수
  • throughputPerHour — 시간당 처리량
  • leadTime / waitingTime / deliveryTime — p50, p95, mean + histogram

4-Stage Order Lifecycle

pickupApproach — 픽업 지점까지 이동 └─ moveToPickup → pickupArrive loading — 적재 (대기 + 실행) └─ pickupStart → pickupDone dropApproach — 하차 지점까지 이동 └─ moveToDrop → dropArrive unloading — 하차 (대기 + 실행) └─ dropStart → dropDone 각 단계별 평균 시간 산출 (mean)

Stop Reason (Bitmask 기반 6종)

Bit 0: SENSOR_COLLISION   — 센서 충돌로 정지
Bit 1: LOCK_WAIT          — 합류점 Lock 대기
Bit 2: LOADING             — 적재 중 정지
Bit 3: UNLOADING           — 하차 중 정지
Bit 4: PRE_BRAKE           — 사전 감속 정지
Bit 5: PATH_BLOCKED        — 경로 차단됨

+ sensor collision count, path remaining 추가 집계

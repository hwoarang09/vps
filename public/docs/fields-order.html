2. 필드별 상세

ORDER_ID Offset 31

항목설명
용도현재 차량에 할당된 Transfer 주문의 고유 ID. 0이면 주문이 없는 상태 (IDLE).
WriterAutoMgr.update() — Transfer assign 단계에서 nextOrderId++ 값을 기록. UNLOADING 완료 시 0으로 초기화.
Reader
  • IndividualControlPanel — 선택 차량의 주문 정보 표시
  • AutoMgr.update() (UNLOADING 완료) — 로그 콜백용 orderId 읽기
  • SimLogger.onOrderComplete() — 완료 오더 로그 기록
값 범위0 (없음) 또는 1 이상의 양의 정수. AutoMgr 인스턴스 수명 동안 단조 증가.

ORDER_SRC_STATION Offset 32

항목설명
용도픽업(적재) 스테이션의 인덱스. 오더 할당 시 기록되어 전체 라이프사이클 동안 유지.
WriterAutoMgr.update() Transfer assign — srcStationIdx + 1 (1-based). UNLOADING 완료 시 0으로 초기화.
Reader
  • IndividualControlPanel — 출발 스테이션 표시
  • AutoMgr.update() (UNLOADING 완료) — onOrderComplete 콜백 파라미터
값 범위0 (없음) 또는 1 ~ stations.length (1-based station index)

ORDER_DEST_STATION Offset 33

항목설명
용도드롭(하역) 스테이션의 인덱스. 오더 할당 시 기록.
WriterAutoMgr.update() Transfer assign — destStationIdx + 1 (1-based). UNLOADING 완료 시 0으로 초기화.
Reader
  • IndividualControlPanel — 도착 스테이션 표시
  • AutoMgr.update() (UNLOADING 완료) — onOrderComplete 콜백 파라미터
값 범위0 (없음) 또는 1 ~ stations.length (1-based station index)

ORDER_DEST_EDGE Offset 34

항목설명
용도오더 목적지 edge ID. LogicData.DESTINATION_EDGE와 연동되어 라우팅 대상 edge를 나타냄. applyPathToVehicle()에서 DESTINATION_EDGE에 기록되며, 이 필드는 오더 컨텍스트에서 별도 보존.
WriterAutoMgr.applyPathToVehicle() 내부에서 LogicData.DESTINATION_EDGE에 기록하며, OrderData 영역은 현재 코드에서 직접 쓰지 않음 (예약 필드). 실질적 목적지는 DESTINATION_EDGE(offset 24)가 관리.
Reader현재 직접 읽는 UI/로직 없음. 향후 확장용 예약 필드.
값 범위0 (없음) 또는 1-based edge index

MOVE_TO_PICKUP_TS Offset 35

항목설명
용도오더 생성(=차량 배정) 시각. 반송 전체 리드타임의 기준점(t0).
WriterAutoMgr.update() Transfer assign — data[ptr + OrderData.MOVE_TO_PICKUP_TS] = simulationTime
Reader
  • AutoMgr.update() UNLOADING 완료 — leadTime, waitingTime 산출
  • onOrderComplete 콜백 — createTs, assignTs
  • 4-stage 평균 — pickupApproachSum += pickupArriveTs - moveToPickupTs
값 범위simulationTime (ms). 시뮬레이션 시작 이후 경과 시간.

PICKUP_ARRIVE_TS Offset 36

항목설명
용도차량이 픽업 스테이션에 도착(정차)한 시각. Pickup Approach 구간 종료 시점.
WriterAutoMgr.update() MOVE_TO_LOAD 분기 — 차량이 src station edge 위에서 정차하고 EDGE_RATIO >= TARGET_RATIO일 때:
data[ptr + OrderData.PICKUP_ARRIVE_TS] = simulationTime
Reader
  • UNLOADING 완료 — pickupApproachSum += pickupArriveTs - moveToPickupTs
  • UNLOADING 완료 — loadingSum += pickupDoneTs - pickupArriveTs
  • onOrderComplete 콜백 — pickupStartTs
값 범위simulationTime (ms)

PICKUP_START_TS Offset 37

항목설명
용도적재(Loading) 시작 시각. 현재 구현에서는 PICKUP_ARRIVE_TS와 동일 시점에 기록됨 (도착 즉시 적재 시작).
WriterAutoMgr.update() MOVE_TO_LOAD 분기 — LOADING 전환 시:
data[ptr + OrderData.PICKUP_START_TS] = simulationTime
Reader현재 직접 소비하는 로직 없음. PICKUP_ARRIVE_TS와 동일 시점이므로 4-stage 계산에서는 PICKUP_ARRIVE_TS를 사용.
값 범위simulationTime (ms). 항상 PICKUP_ARRIVE_TS와 동일.

PICKUP_DONE_TS Offset 38

항목설명
용도적재 완료 시각. dwell timer(dwellMs, 기본 7초) 소진 후 기록. waitingTime / deliveryTime 경계점.
WriterAutoMgr.update() LOADING complete 분기 — now >= dwellTimers.get(vehId) 충족 시:
data[ptr + OrderData.PICKUP_DONE_TS] = simulationTime
Reader
  • UNLOADING 완료 — waitingTimes.push(pickupDoneTs - moveToPickupTs)
  • UNLOADING 완료 — deliveryTimes.push(simulationTime - pickupDoneTs)
  • 4-stage — loadingSum += pickupDoneTs - pickupArriveTs
  • 4-stage — dropApproachSum += dropArriveTs - pickupDoneTs
  • onOrderComplete 콜백 — pickupCompleteTs
값 범위simulationTime (ms). 항상 PICKUP_ARRIVE_TS + dwellMs 이상.

MOVE_TO_DROP_TS Offset 39

항목설명
용도드롭 스테이션으로 이동 시작 시각. 현재 구현에서 PICKUP_DONE_TS와 동일 시점에 기록 (적재 완료 즉시 출발).
WriterAutoMgr.update() LOADING complete 분기 — MOVE_TO_UNLOAD 전환 시:
data[ptr + OrderData.MOVE_TO_DROP_TS] = simulationTime
Reader현재 직접 소비하는 로직 없음. PICKUP_DONE_TS와 동일 시점.
값 범위simulationTime (ms). 항상 PICKUP_DONE_TS와 동일.

DROP_ARRIVE_TS Offset 40

항목설명
용도차량이 드롭 스테이션에 도착(정차)한 시각. Drop Approach 구간 종료 시점.
WriterAutoMgr.update() MOVE_TO_UNLOAD 분기 — 차량이 dest station edge 위에서 정차하고 EDGE_RATIO >= TARGET_RATIO일 때:
data[ptr + OrderData.DROP_ARRIVE_TS] = simulationTime
Reader
  • UNLOADING 완료 — dropApproachSum += dropArriveTs - pickupDoneTs
  • UNLOADING 완료 — unloadingSum += simulationTime - dropArriveTs
  • onOrderComplete 콜백 — dropStartTs
값 범위simulationTime (ms)

DROP_START_TS Offset 41

항목설명
용도하역(Unloading) 시작 시각. 현재 구현에서 DROP_ARRIVE_TS와 동일 시점에 기록.
WriterAutoMgr.update() MOVE_TO_UNLOAD 분기 — UNLOADING 전환 시:
data[ptr + OrderData.DROP_START_TS] = simulationTime
Reader현재 직접 소비하는 로직 없음. DROP_ARRIVE_TS와 동일 시점.
값 범위simulationTime (ms). 항상 DROP_ARRIVE_TS와 동일.

DROP_DONE_TS Offset 42

항목설명
용도하역 완료 시각. dwell timer 소진 후 기록. 전체 반송 리드타임의 종료 시점.
WriterAutoMgr.update() UNLOADING complete 분기 — now >= dwellTimers.get(vehId) 충족 시:
data[ptr + OrderData.DROP_DONE_TS] = simulationTime
Reader
  • 바로 다음 줄에서 leadTime, deliveryTime 산출에 사용 (simulationTime 자체가 곧 DROP_DONE_TS)
  • 4-stage — unloadingSum += simulationTime - dropArriveTs
값 범위simulationTime (ms). 항상 DROP_ARRIVE_TS + dwellMs 이상.

3. Transfer State Machine

Transfer 반송은 JobState enum으로 관리되는 6단계 상태 머신입니다. AutoMgr.update()가 매 프레임 transferringVehicles set을 순회하며 상태를 전이합니다.

상태 전이 다이어그램

IDLE (1) │ │ Transfer assign: shouldAssignTransfer() && isSwappable() │ WRITE: ORDER_ID, ORDER_SRC_STATION, ORDER_DEST_STATION, MOVE_TO_PICKUP_TS │ ▼ MOVE_TO_LOAD (2) │ │ 차량이 src station edge 진입 시: │ - preloadNextPath() → dest 경로 미리 적재 │ - TARGET_RATIO = srcStation.ratio (station 위치에서 정차) │ │ 차량 정차 + EDGE_RATIO >= TARGET_RATIO: │ WRITE: PICKUP_ARRIVE_TS, PICKUP_START_TS │ dwellTimer 시작 (기본 7초) │ ▼ LOADING (3) │ │ dwellTimer 만료 (now >= dwellTimers.get(vehId)): │ WRITE: PICKUP_DONE_TS, MOVE_TO_DROP_TS │ TARGET_RATIO = 1, MOVING_STATUS = MOVING (출발) │ ▼ MOVE_TO_UNLOAD (4) │ │ 차량이 dest station edge 진입 시: │ - preloadLoopPath() → 하역 후 순환 경로 미리 적재 │ - TARGET_RATIO = destStation.ratio │ - dwellTimer = Infinity (sentinel, 재진입 방지) │ │ 차량 정차 + EDGE_RATIO >= TARGET_RATIO: │ WRITE: DROP_ARRIVE_TS, DROP_START_TS │ dwellTimer 시작 (기본 7초) │ ▼ UNLOADING (5) │ │ dwellTimer 만료: │ WRITE: DROP_DONE_TS │ KPI 기록 (leadTimes, waitingTimes, deliveryTimes, 4-stage) │ onOrderComplete 콜백 호출 │ CLEAR: ORDER_ID=0, SRC_STATION=0, DEST_STATION=0 │ TARGET_RATIO=1, MOVING_STATUS=MOVING │ transferringVehicles.delete(vehId) │ ▼ IDLE (1) ← 다음 오더 대기

주요 조건 상세

전이조건비고
IDLE → MOVE_TO_LOAD transferEnabled && shouldAssignTransfer() (utilization/throughput 할당량) && isSwappable() (미결 lock 없음) src/dest station 랜덤 선택. assignToStation()으로 src까지 경로 적용.
MOVE_TO_LOAD → LOADING isStopped && EDGE_RATIO >= TARGET_RATIO && pendingSrcStation 해소됨 lock/충돌로 station 직전에 멈춘 경우를 도착과 구분하기 위해 ratio 비교 필수.
LOADING → MOVE_TO_UNLOAD now >= dwellTimers.get(vehId) dwell 시간은 dwellMs (기본 7000ms). preloadNextPath로 이미 경로 적재됨.
MOVE_TO_UNLOAD → UNLOADING isStopped && currentEdge === destStation.edgeIndex && EDGE_RATIO >= TARGET_RATIO dest station edge 진입 시 preloadLoopPath로 순환 경로 미리 적재.
UNLOADING → IDLE now >= dwellTimers.get(vehId) KPI 기록 후 오더 데이터 전부 초기화. preloadLoopPath 경로로 자연스럽게 이동 재개.

4. Timestamp 기록 시점

8개 타임스탬프는 모두 simulationTime (ms) 단위입니다. 시뮬레이션 시작(또는 리셋) 이후 경과한 밀리초 값이며, 실시간 벽시계(Date.now)가 아닙니다.

타임라인

t0 MOVE_TO_PICKUP_TS ─── Transfer assign │ │ [Pickup Approach: 차량이 src station으로 이동] │ t1 PICKUP_ARRIVE_TS ──── src station 도착 (정차 + ratio 충족) PICKUP_START_TS ───── (동일 시점, 도착 즉시 적재 시작) │ │ [Loading: dwellMs 대기 (기본 7초)] │ t2 PICKUP_DONE_TS ────── dwell timer 만료 MOVE_TO_DROP_TS ───── (동일 시점, 적재 완료 즉시 출발) │ │ [Drop Approach: 차량이 dest station으로 이동] │ t3 DROP_ARRIVE_TS ────── dest station 도착 (정차 + ratio 충족) DROP_START_TS ─────── (동일 시점, 도착 즉시 하역 시작) │ │ [Unloading: dwellMs 대기 (기본 7초)] │ t4 DROP_DONE_TS ──────── dwell timer 만료 → KPI 기록 + 오더 초기화

참고: 현재 구현에서는 "도착 즉시 작업 시작" 모델이므로 PICKUP_ARRIVE_TS == PICKUP_START_TS, DROP_ARRIVE_TS == DROP_START_TS, PICKUP_DONE_TS == MOVE_TO_DROP_TS입니다. 향후 대기열(queue) 로직이 추가되면 이 타임스탬프들이 분리될 수 있습니다.

5. KPI 산출

오더 완료(UNLOADING complete) 시점에 AutoMgr.orderStats에 3가지 시간 지표와 4-stage 분해를 누적합니다.

3대 시간 지표

KPI산식의미
leadTime DROP_DONE_TS - MOVE_TO_PICKUP_TS
(= t4 - t0)		 오더 생성부터 하역 완료까지 전체 소요 시간
waitingTime PICKUP_DONE_TS - MOVE_TO_PICKUP_TS
(= t2 - t0)		 오더 생성부터 적재 완료까지. Pickup Approach + Loading 합산.
deliveryTime DROP_DONE_TS - PICKUP_DONE_TS
(= t4 - t2)		 적재 완료부터 하역 완료까지. Drop Approach + Unloading 합산.

관계식: leadTime = waitingTime + deliveryTime

4-Stage 분해

각 stage의 합산값(sum)과 오더 수(stageCount)를 누적하여, flush 시점에 평균을 산출합니다.

Stage산식구간
pickupApproach PICKUP_ARRIVE_TS - MOVE_TO_PICKUP_TS
(= t1 - t0)		 src station까지 이동 시간
loading PICKUP_DONE_TS - PICKUP_ARRIVE_TS
(= t2 - t1)		 적재 대기 시간 (dwellMs)
dropApproach DROP_ARRIVE_TS - PICKUP_DONE_TS
(= t3 - t2)		 dest station까지 이동 시간
unloading DROP_DONE_TS - DROP_ARRIVE_TS
(= t4 - t3)		 하역 대기 시간 (dwellMs)

관계식: waitingTime = pickupApproach + loading  |  deliveryTime = dropApproach + unloading

4-Stage 평균 산출 (FabContext.flushOrderStats)

// sum / count / 1000 → 초 단위 평균 pickupApproachMean = pickupApproachSum / stageCount / 1000 loadingMean = loadingSum / stageCount / 1000 dropApproachMean = dropApproachSum / stageCount / 1000 unloadingMean = unloadingSum / stageCount / 1000

6. orderStats 히스토그램

FabContext.flushOrderStats()에서 raw 배열을 정렬 후 히스토그램 + 백분위수를 산출합니다.

히스토그램 설정

지표Bucket 폭Bucket 수커버 범위오버플로우
leadTime 10초 (LEAD_BUCKET_SEC) 37 0 ~ 360초 (6분) 마지막 bucket [36] = 360초 이상
waitingTime 5초 (WAITING_BUCKET_SEC) 37 0 ~ 180초 (3분) 마지막 bucket [36] = 180초 이상
deliveryTime 5초 (DELIVERY_BUCKET_SEC) 37 0 ~ 180초 (3분) 마지막 bucket [36] = 180초 이상

히스토그램 생성 알고리즘

function summarize(raw: number[], bucketSec: number) { // 1. 정렬 (p50/p95 산출용) const sorted = raw.slice().sort((a, b) => a - b); // 2. 히스토그램 빌드 (37 buckets) const histogram = new Array(37).fill(0); for (const val of sorted) { const sec = val / 1000; // ms → sec 변환 const idx = Math.min(36, Math.floor(sec / bucketSec)); histogram[idx]++; } // 3. 백분위수 + 평균 const p50 = percentileSorted(sorted, 0.50) / 1000; // 초 단위 const p95 = percentileSorted(sorted, 0.95) / 1000; const mean = sum(sorted) / sorted.length / 1000; return { p50, p95, mean, histogram }; }

백분위수 산출

function percentileSorted(sorted: number[], p: number): number { if (sorted.length === 0) return 0; const idx = Math.ceil(p * sorted.length) - 1; return sorted[Math.max(0, idx)]; } // 예시: 100개 샘플에서 p50 → idx = ceil(50) - 1 = 49번째 // 100개 샘플에서 p95 → idx = ceil(95) - 1 = 94번째

Worker → Main Thread 전달

flushOrderStats()는 주기적으로 postMessage({ type: "ORDER_STATS", ... })를 호출하여 아래 데이터를 Main Thread로 전송합니다:

필드타입설명
completednumber완료된 오더 수
throughputPerHournumber시간당 처리량 (completed / elapsed * 3,600,000)
leadTimeP50 / P95 / Meannumber리드타임 백분위수 및 평균 (초)
leadTimeHistogramnumber[37]리드타임 히스토그램 (10초 bucket)
waitingTime*동일 구조대기시간 (5초 bucket)
deliveryTime*동일 구조배송시간 (5초 bucket)
pickupApproachMeannumberPickup Approach 평균 (초)
loadingMeannumberLoading 평균 (초)
dropApproachMeannumberDrop Approach 평균 (초)
unloadingMeannumberUnloading 평균 (초)

Main Thread에서는 useOrderStatsStore (Zustand)가 fab별로 수신하며, FLUSH_INTERVAL_MS = 5000ms 간격으로 throttle-batch하여 React re-render를 최소화합니다.

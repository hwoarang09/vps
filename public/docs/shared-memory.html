Shared Memory LayoutSharedArrayBuffer 기반 zero-copy 차량 데이터 통신

1. 개요

VPS는 SharedArrayBuffer를 사용하여 Worker(시뮬레이션)와 Main Thread(렌더링) 사이에 zero-copy로 데이터를 교환합니다. postMessage 직렬화 없이, 양쪽 스레드가 동일한 메모리 버퍼를 직접 읽고 씁니다.

  • Worker(WRITE) → Main Thread(READ ONLY) 단방향 패턴
  • 차량 위치, 상태, 경로, 센서 등 모든 고빈도 데이터는 공유 메모리를 통해 전달
  • Atomics.store / Atomics.load로 동기화 플래그 관리

핵심 파일

FileRole
src/common/vehicle/constants.ts필드 오프셋, enum, 버퍼 크기 상수 정의
src/shmSimulator/MemoryLayoutManager.tsFab별 메모리 할당 및 region 관리

데이터 흐름

Worker (SimulationEngine) │ ├─ movement.step() → vehicleDataArray[offset + X] = newX ├─ collision.step() → vehicleDataArray[offset + HIT_ZONE] = zone ├─ routing.step() → pathBuffer[vehIdx] = [len, e0, e1, ...] └─ checkpoint.update() → checkpointBuffer[vehIdx] = [count, ...] │ ↓ SharedArrayBuffer (zero-copy) │ Main Thread (useFrame loop) │ ├─ VehicleRenderer → reads vehicleDataArray → InstancedMesh.setMatrixAt() ├─ SensorRenderer → reads sensorPointBuffer → line geometry └─ PathOverlay → reads pathBuffer → debug visualization

2. Vehicle Data Array

차량 1대당 43 fields, 각 필드 Float32(4 bytes) = 172 bytes/vehicle.
전체 배열은 Float32Array 뷰로 접근하며, 오프셋은 vehicleIndex * VEHICLE_DATA_SIZE + fieldOffset으로 계산합니다.

4. Checkpoint Buffer

차량당 801 floats의 체크포인트 큐. Float32Array 기반.

MAX_CHECKPOINTS_PER_VEHICLE = 200
CHECKPOINT_FIELDS = 4 (edge, ratio, flags, target)
차량당 크기: 1 + (200 × 4) = 801 floats = 3,204 bytes

메모리 레이아웃

checkpointBuffer[vehIdx * 801 + 0]   = count          // 현재 체크포인트 수
checkpointBuffer[vehIdx * 801 + 1]   = cp[0].edge      // 첫 번째 CP edge
checkpointBuffer[vehIdx * 801 + 2]   = cp[0].ratio     // 첫 번째 CP ratio
checkpointBuffer[vehIdx * 801 + 3]   = cp[0].flags     // 첫 번째 CP flags (bitmask)
checkpointBuffer[vehIdx * 801 + 4]   = cp[0].target    // 첫 번째 CP target
checkpointBuffer[vehIdx * 801 + 5]   = cp[1].edge      // 두 번째 CP edge
...
checkpointBuffer[vehIdx * 801 + 797] = cp[199].edge    // 마지막 CP edge
checkpointBuffer[vehIdx * 801 + 798] = cp[199].ratio
checkpointBuffer[vehIdx * 801 + 799] = cp[199].flags
checkpointBuffer[vehIdx * 801 + 800] = cp[199].target

일반 공식

// n번째 체크포인트 (0-indexed)의 field 접근
const base = vehIdx * 801;
const cpOffset = base + 1 + n * 4;
const edge   = checkpointBuffer[cpOffset + 0];
const ratio  = checkpointBuffer[cpOffset + 1];
const flags  = checkpointBuffer[cpOffset + 2];
const target = checkpointBuffer[cpOffset + 3];

5. Path Buffer

차량당 101 Int32s. Int32Array 기반 경로 데이터.

MAX_PATH_LENGTH = 100
차량당 크기: 1 + 100 = 101 Int32s = 404 bytes

메모리 레이아웃

pathBuffer[vehIdx * 101 + 0]  = pathLen        // 현재 경로 길이 (0 ~ 100)
pathBuffer[vehIdx * 101 + 1]  = edge[0]        // 첫 번째 엣지
pathBuffer[vehIdx * 101 + 2]  = edge[1]        // 두 번째 엣지
...
pathBuffer[vehIdx * 101 + N]  = edge[N-1]      // N번째 엣지
// pathLen 이후의 슬롯은 미사용 (stale 데이터 가능)
// 차량 i의 경로 읽기
const base = i * 101;
const len = pathBuffer[base];
for (let j = 0; j < len; j++) {
  const edgeId = pathBuffer[base + 1 + j];
}

7. Render Buffer (GPU 최적화)

렌더링 전용 버퍼는 Worker 데이터에서 GPU 전송에 필요한 최소한의 데이터만 추출하여 별도로 유지합니다.

Vehicle Render Buffer

차량당 4 floats: [x, y, z, rotation_deg]
16 bytes/vehicle — InstancedMesh matrix 계산에 필요한 최소 데이터

// renderBuffer layout (Float32Array)
renderBuffer[i * 4 + 0] = x;             // world X
renderBuffer[i * 4 + 1] = y;             // world Y
renderBuffer[i * 4 + 2] = z;             // world Z
renderBuffer[i * 4 + 3] = rotation_deg;  // rotation in degrees

Sensor Render Buffer

차량당 28 floats: 7 sections × 4 floats (x1, y1, x2, y2)
센서 영역 경계선을 line segment로 표현. 112 bytes/vehicle.

Multi-Fab Continuous Layout

다중 fab 환경에서 렌더 버퍼는 연속 배치(concatenated)됩니다.

// Multi-fab render buffer layout
[Fab0: veh0, veh1, ..., vehN₀] [Fab1: veh0, veh1, ..., vehN₁] [Fab2: ...]
 └── offset = 0                 └── offset = N₀ * 4             └── ...

Render buffer vs Worker buffer 분리 이유: Worker 버퍼는 43 floats/vehicle 전체를 포함하지만, GPU 전송에는 위치+회전 4 floats만 필요. 별도 버퍼로 분리하면 bufferSubData() 호출 시 불필요한 데이터 전송을 방지합니다.

8. Memory Layout Manager

MemoryLayoutManager는 Fab 단위로 SharedArrayBuffer를 할당하고, 각 region의 오프셋과 크기를 관리합니다.

Per-Fab 할당 구조

9. 메모리 효율 분석

SharedArrayBuffer vs postMessage 비교

방식1,000 vehicles / frameOverheadLatency
postMessage (structured clone) ~172 KB 복사 + 직렬화 HIGH GC pressure, 메모리 2배 ~2-5 ms
Transferable (ArrayBuffer transfer) 0 bytes 복사 (소유권 이전) MED 원본 무효화, 재할당 필요 ~0.1 ms
SharedArrayBuffer 0 bytes 복사 (공유) NONE 추가 메모리 없음 ~0 ms (즉시)

GPU Transfer 최적화

Render buffer: 4n floats = 16n bytes/frame

만약 전체 transformation matrix를 전송한다면 16 floats × 4 bytes = 64n bytes/frame. Render buffer는 position + rotation만 전송하고, matrix 조립은 GPU(vertex shader)에서 수행하여 대역폭을 75% 절감합니다.

방식Data/vehicle1,000 veh @ 60fps
Full matrix transfer64 bytes3.84 MB/s
Render buffer (x,y,z,rot)16 bytes0.96 MB/s

Cache Coherency: Section-Based Layout

동일 종류의 데이터를 연속 메모리에 배치하여 CPU 캐시 히트율을 높입니다.

  • AoS (Array of Structs) 방식이면 차량 전체 데이터가 연속 → 특정 필드만 읽을 때 캐시 낭비
  • VPS의 region 분리: vehicleRegion / sensorRegion / pathRegion / checkpointRegion 각각 연속 배치
  • 렌더링 시 vehicleRegion의 X, Y, Z, ROTATION만 순차 접근 → 캐시 라인 효율 극대화
// 캐시 친화적 순차 접근 패턴
for (let i = 0; i < vehicleCount; i++) {
  const base = i * VEHICLE_DATA_SIZE;
  matrix.compose(
    view[base + 0],  // X  (연속 메모리)
    view[base + 1],  // Y
    view[base + 2],  // Z
    view[base + 3]   // ROTATION
  );
}

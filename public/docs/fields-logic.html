LogicData 필드별 상세Offset 21 ~ 30 — Lock / Traffic / Job / Checkpoint 상태

상세 로직 — Flag 처리 순서

processCheckpointFlags()의 처리 순서는 교착 방지를 위해 엄격히 지정됩니다:

  1. MOVE_PREPARE (0x08) — NEXT_EDGE 배열 채우기 (가장 먼저, edge 준비 필수)
  2. LOCK_RELEASE (0x04) — 이전 merge의 lock 해제 (다음 차량 grant)
  3. LOCK_REQUEST (0x01) — 새 merge의 lock 요청
  4. LOCK_WAIT (0x02) — lock 획득 대기 (미획득 시 flag 유지, 정차)
// checkpoint-processor.ts: flag 순서대로 처리
if (cpFlags & CheckpointFlags.MOVE_PREPARE) {
    handleMovePrepare(vehicleId, state);           // NEXT_EDGE 채우기
    cpFlags &= ~CheckpointFlags.MOVE_PREPARE;      // 비트 클리어
}
if (cpFlags & CheckpointFlags.LOCK_RELEASE) {
    handleLockRelease(vehicleId, state, eName);    // lock 해제
    cpFlags &= ~CheckpointFlags.LOCK_RELEASE;
}
if (cpFlags & CheckpointFlags.LOCK_REQUEST) {
    handleLockRequest(vehicleId, state);           // lock 요청
    cpFlags &= ~CheckpointFlags.LOCK_REQUEST;      // 요청 후 즉시 클리어
}
if (cpFlags & CheckpointFlags.LOCK_WAIT) {
    const granted = handleLockWait(vehicleId, state);
    if (granted) {
        cpFlags &= ~CheckpointFlags.LOCK_WAIT;     // 획득 시만 클리어
    }
    // 미획득 시 flag 유지 → 다음 프레임에서 재시도
}

LOCK_WAIT 특수 동작: 다른 flag와 달리 LOCK_WAIT는 lock을 획득해야만 비트가 클리어됩니다. 미획득 시 flag가 유지되어 다음 프레임에서 재시도하며, 차량은 STOPPED 상태로 정차합니다.

    const ptr = (startIdx + i) * VEHICLE_DATA_SIZE;

    // JOB_STATE: 값 비교로 분류
    const js = fullData[ptr + LogicData.JOB_STATE];
    if (js === JobState.IDLE) stats.jobIdle++;
    else if (js === JobState.MOVE_TO_LOAD) stats.jobMoveToLoad++;
    // ... (7개 상태 전수 분류)

    // TRAFFIC_STATE: 값 비교로 분류
    const ts = fullData[ptr + LogicData.TRAFFIC_STATE];
    if (ts === TrafficState.FREE) stats.trafficFree++;
    // ...

    // STOP_REASON: 비트마스크 AND로 복수 사유 동시 카운트
    const sr = fullData[ptr + LogicData.STOP_REASON];
    if (sr & StopReason.LOCKED) stats.stopLocked++;
    if (sr & StopReason.SENSORED) stats.stopSensored++;
    // ...

    // PATH_REMAINING: 합산 및 최대값
    const pathRem = fullData[ptr + LogicData.PATH_REMAINING];
    totalPath += pathRem;
    if (pathRem > stats.maxPathRemaining) stats.maxPathRemaining = pathRem;
}

Checkpoint 필드 라이프사이클 요약

4개의 CP 필드(CURRENT_CP_EDGE, CURRENT_CP_RATIO, CURRENT_CP_FLAGS, CURRENT_CP_TARGET)는 하나의 단위로 동작합니다.

Phase 1: 경로 할당 TransferMgr.assignCommand() └─ checkpointArray에 CP 배열 기록 └─ CHECKPOINT_HEAD = 0 └─ CURRENT_CP_EDGE = 0 (로드 트리거) Phase 2: CP 로드 (ensureCheckpointLoaded) CURRENT_CP_EDGE === 0? YES → loadNextCheckpoint() ├─ checkpointArray[head] → CURRENT_CP_EDGE / RATIO / FLAGS / TARGET └─ CHECKPOINT_HEAD++ Phase 3: 도달 대기 (checkCheckpointReached) 매 프레임: ├─ currentEdge === CP_EDGE && currentRatio ≥ CP_RATIO → HIT ├─ currentEdge !== CP_EDGE && isCpEdgeBehind() → MISSED └─ otherwise → WAITING Phase 4: Flag 처리 (processCheckpointFlags) HIT 시: MOVE_PREPARE → LOCK_RELEASE → LOCK_REQUEST → LOCK_WAIT 각 flag 처리 후 비트 클리어 (LOCK_WAIT 미획득 시 flag 유지 → 재시도) MISSED 시: handleMissedCheckpoint() → 동일 핸들러 호출 (WAIT 미획득 시 CP 유지) flags = 0 강제 클리어 → 다음 CP 로드 Phase 5: 다음 CP 로드 flags === 0 → loadNextCheckpoint() ├─ head < count → Phase 2 반복 └─ head ≥ count → CP 소진 (CURRENT_CP_EDGE = 0) Catch-up: 짧은 edge를 한 프레임에 통과 시 최대 10개 CP 연속 처리

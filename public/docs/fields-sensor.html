SensorData 필드 상세Offset 18 ~ 20 — 센서 프리셋, 충돌 감지 영역, 충돌 대상 차량

1. 개요

SensorData 섹션(3개 필드)은 라이다 센서 기반 충돌 감지 시스템의 상태를 저장합니다. 차량이 주행하는 Edge 타입(직선/커브/합류/분기)에 따라 센서 프리셋이 결정되고, 매 시뮬레이션 스텝마다 앞차와의 충돌 판정 결과(HitZone)와 대상 차량 ID가 기록됩니다.

OffsetFieldType요약
18PRESET_IDXFloat32적용 중인 센서 프리셋 인덱스 (PresetIndex enum)
19HIT_ZONEFloat32충돌 감지 영역 (HitZone enum)
20COLLISION_TARGETFloat32충돌 대상 차량 ID (-1 = 없음)

값 범위 — PresetIndex enum

값이름설명센서 특성
0STRAIGHT직진 구간정면으로 길게 뻗음 (approach 4.5m, 좌우 각도 0°)
1CURVE_LEFT좌회전 구간왼쪽으로 치우침 (좌 65°, 우 -30°, 길이 1m)
2CURVE_RIGHT우회전 구간오른쪽으로 치우침 (좌 -30°, 우 -65°, 길이 1m)
3U_TURN유턴 (180°) 구간좌우 대칭 넓은 각도 (좌 45°, 우 -45°, 길이 1m)
4MERGE합류 구간넓은 시야각 (좌 25°, 우 -25°, approach 1.8m)
5BRANCH분기 구간넓은 시야각 (좌 30°, 우 -30°, approach 1.8m)

상세 로직 — Edge 전환 시 프리셋 결정

updateSensorPresetForEdge() 함수가 Edge의 vos_rail_type 속성을 분석하여 프리셋을 결정합니다.

// edgeTransition.ts — updateSensorPresetForEdge()
const railType = edge.vos_rail_type;
const isCurve = railType?.includes("CURVE") || railType?.startsWith("C");

if (railType === EdgeType.CURVE_180)
    presetIdx = PresetIndex.U_TURN;           // 3
else if (railType === EdgeType.LEFT_CURVE || (isCurve && edge.curve_direction === "left"))
    presetIdx = PresetIndex.CURVE_LEFT;       // 1
else if (railType === EdgeType.RIGHT_CURVE || (isCurve && edge.curve_direction === "right"))
    presetIdx = PresetIndex.CURVE_RIGHT;      // 2
else
    presetIdx = PresetIndex.STRAIGHT;         // 0

data[ptr + SensorData.PRESET_IDX] = presetIdx;

참고: 현재 MERGE(4)와 BRANCH(5)는 Edge 전환 로직에서 자동으로 설정되지 않습니다. 이 프리셋은 커스텀 설정 또는 별도 로직에서 수동으로 할당해야 합니다. 기본적으로 합류/분기 Edge도 STRAIGHT가 적용됩니다.

상세 로직 — HitZone이 차량 행동에 미치는 영향

// collisionCommon.ts — applyCollisionZoneLogic()

if (hitZone === HitZone.NONE) {
    data[ptrBack + SensorData.COLLISION_TARGET] = -1;
    data[ptrBack + MovementData.DECELERATION] = 0;
    // STOPPED 상태였으면 MOVING으로 복원
    if (data[ptrBack + MovementData.MOVING_STATUS] === MovingStatus.STOPPED)
        data[ptrBack + MovementData.MOVING_STATUS] = MovingStatus.MOVING;
    return;
}

data[ptrBack + SensorData.COLLISION_TARGET] = targetVehId;

if (hitZone === HitZone.STOP) {
    // 즉시 정지 (감속도 -Infinity)
    data[ptrBack + MovementData.MOVING_STATUS] = MovingStatus.STOPPED;
    data[ptrBack + MovementData.VELOCITY] = 0;
    data[ptrBack + MovementData.DECELERATION] = 0;
    return;
}

// APPROACH 또는 BRAKE
const isBrake = hitZone === HitZone.BRAKE;
const minSpeed = isBrake ? brakeMinSpeed : approachMinSpeed;
const decValue = isBrake ? preset.zones.brake.dec : preset.zones.approach.dec;

if (velocity > minSpeed)
    data[ptrBack + MovementData.DECELERATION] = decValue;
else
    data[ptrBack + MovementData.DECELERATION] = 0;

5. 센서 프리셋 시스템

프리셋 구조 (SensorPreset)

각 프리셋은 3개의 존(approach/brake/stop)에 대해 센서 각도, 길이, 감속도를 정의합니다.

interface SensorZone {
    leftAngle: number;    // 왼쪽 센서 각도 (degree, 정면=0)
    rightAngle: number;   // 오른쪽 센서 각도 (degree)
    leftLength: number;   // 왼쪽 센서 길이 (meters)
    rightLength: number;  // 오른쪽 센서 길이 (meters)
    dec: number;          // 감속도 (m/s², 음수)
}

interface SensorPreset {
    zones: {
        approach: SensorZone;
        brake: SensorZone;
        stop: SensorZone;
    };
    leftAngle: number;    // 전체 프리셋 대표 각도
    rightAngle: number;
    leftLength: number;
    rightLength: number;
}

6. 센서 포인트 버퍼 (SensorPointArray)

PRESET_IDX와 차량 위치/회전을 바탕으로 계산된 센서 사각형의 월드 좌표를 저장하는 별도 버퍼입니다. Vehicle Data Array와는 별개의 Float32Array이며, SAT 충돌 검사에서 직접 참조됩니다.

메모리 레이아웃

차량 1대당 36 floats (144 bytes) = 3 zones × 12 floats/zone

SensorPointArray 레이아웃 (per vehicle)
+---------------------------------------------------+
| Zone 0 (APPROACH): 12 floats (offset 0~11)        |
| Zone 1 (BRAKE):    12 floats (offset 12~23)       |
| Zone 2 (STOP):     12 floats (offset 24~35)       |
+---------------------------------------------------+

각 Zone 내부 (12 floats = 6 points × 2 좌표):
  FL_X(0) FL_Y(1)  — 차량 앞 왼쪽 모서리
  FR_X(2) FR_Y(3)  — 차량 앞 오른쪽 모서리
  BL_X(4) BL_Y(5)  — 차량 뒤 왼쪽 모서리
  BR_X(6) BR_Y(7)  — 차량 뒤 오른쪽 모서리
  SL_X(8) SL_Y(9)  — 센서 왼쪽 끝 (프리셋 기반)
  SR_X(10) SR_Y(11) — 센서 오른쪽 끝 (프리셋 기반)

포인트 계산 흐름

updateSensorPoints() 함수가 매 프레임 commitVehicleState()에서 호출됩니다.

  1. 본체 모서리 (FL/FR/BL/BR): 차량 중심 (x, y)에서 bodyLength/2 만큼 전후, bodyWidth/2 만큼 좌우로 회전 변환하여 4개 모서리 좌표를 계산합니다.
  2. 센서 끝 (SL/SR): 프리셋의 존별 각도(leftAngle/rightAngle)와 길이(leftLength/rightLength)를 로컬 좌표로 변환한 뒤, 차량 회전에 맞춰 월드 좌표로 변환합니다. FL 기준으로 왼쪽 센서(SL), FR 기준으로 오른쪽 센서(SR)의 끝점을 계산합니다.
차량 위에서 본 센서 포인트 배치 SL ---------.------------- SR 센서 끝 / \ / \ / \ FL ----------------------------- FR 차량 앞면 | | | Vehicle | | (본체) | | | BL ----------------------------- BR 차량 뒷면 ^ 주행 방향 센서 사각형 (충돌 검사용): FL -> SL -> SR -> FR 본체 사각형 (타겟 검사용): FL -> BL -> BR -> FR

7. 충돌 감지 알고리즘

하이브리드 전략

VPS는 성능과 정확도를 양립하기 위해 Edge 타입에 따라 서로 다른 충돌 감지 방식을 사용합니다.

Edge 타입방식판정 기준성능
직선 (vos_rail_type 없음)거리 기반축 방향 좌표 차이 vs 프리셋 존별 길이 + bodyLength매우 빠름
커브/합류/분기 (vos_rail_type 있음)SAT 기반센서 사각형과 본체 사각형의 겹침 판정정확하지만 비용 있음

3-Zone 충돌 검사 (SAT 알고리즘)

SAT (Separating Axis Theorem) 원리

두 볼록 다각형(사각형)이 모든 투영 축에서 겹치면 충돌으로 판정합니다. 하나라도 분리되는 축이 있으면 충돌이 아닙니다.

  1. 뒤차의 센서 사각형(FL->SL->SR->FR) 4개 변에 수직인 축 4개에 대해 양쪽 사각형을 투영
  2. 앞차의 본체 사각형(FL->BL->BR->FR) 4개 변에 수직인 축 4개에 대해 양쪽 사각형을 투영
  3. 총 8개 축 모두에서 투영 구간이 겹치면 충돌 확정

Zone 검사 순서

checkSensorCollision()은 STOP(2) -> BRAKE(1) -> APPROACH(0) 순서로 검사합니다. 가장 안쪽(강한 제동) 존부터 검사하여, 충돌이 감지되면 즉시 해당 존 번호를 반환합니다.

// sensorCollision.ts — checkSensorCollision()
for (let zone = 2; zone >= 0; zone--) {
    const sensorBase = baseSensor + zone * SENSOR_POINT_SIZE;
    const targetBase = baseTarget + 0 * SENSOR_POINT_SIZE; // 타겟은 항상 zone 0 (본체)

    if (
        satQuadCheck(data, sensorBase, targetBase, SENSOR_QUAD_IDX, BODY_QUAD_IDX) &&
        satQuadCheck(data, targetBase, sensorBase, BODY_QUAD_IDX, SENSOR_QUAD_IDX)
    ) {
        return zone;  // 2=STOP, 1=BRAKE, 0=APPROACH
    }
}
return -1;  // NONE

Rough Distance Check: SAT 검사 전에 roughDistanceCheck()로 두 차량 간 유클리드 거리가 임계값(~8m) 이내인지 확인합니다. 대부분의 차량 쌍은 이 단계에서 걸러지므로 SAT 호출 횟수가 ~90% 감소합니다.

직선 구간 거리 기반 판정

직선 Edge(vos_rail_type이 없는 경우)에서는 SAT 대신 단순 좌표 차이로 판정합니다.

// verifyFollowingCollision.ts
const axisIdx = edge.axis === "y" ? MovementData.Y : MovementData.X;
const distance = Math.abs(frontPos - backPos);

const stopDist    = preset.zones.stop.leftLength + vehicleLength;
const brakeDist   = preset.zones.brake.leftLength + vehicleLength;
const approachDist = preset.zones.approach.leftLength + vehicleLength;

hitZone = determineLinearHitZone(distance, stopDist, brakeDist, approachDist);

1. Edge 전환 (edgeTransition.ts) │ └─ updateSensorPresetForEdge() └─ data[PRESET_IDX] = presetIdx (Edge 타입에 따라 0~5) 2. 매 프레임 상태 기록 (vehicleState.ts) │ └─ commitVehicleState() └─ updateSensorPoints(sensorPointArray, vehIdx, x, y, rot, presetIdx, config) └─ Zone 0~2 각각에 대해 FL/FR/BL/BR/SL/SR 좌표 계산 └─ sensorPointArray[vehIdx * 36 + zone * 12 + point] 에 기록 3. 충돌 검사 (collision/*.ts) │ ├─ verifyFollowingCollision() — 동일 Edge 앞차 ├─ verifyMergeCollision() — 합류 경쟁 차량 └─ verifyNextPathCollision() — 다음 Edge 꼬리 차량 │ ├─ 직선: determineLinearHitZone(distance, stop, brake, approach) └─ 커브: checkSensorCollision(sensorPointArray, backVeh, frontVeh) └─ SAT(STOP) -> SAT(BRAKE) -> SAT(APPROACH) │ └─ applyCollisionZoneLogic(hitZone, data, ptr, targetVehId) ├─ data[HIT_ZONE] = hitZone ├─ data[COLLISION_TARGET] = targetVehId (or -1) ├─ data[VELOCITY] = 0 (STOP인 경우) ├─ data[DECELERATION] = dec (APPROACH/BRAKE) └─ data[MOVING_STATUS] = STOPPED (STOP인 경우)

9. 성능 최적화

Zero-GC 설계

  • 객체 생성 제로: 센서 포인트 계산과 SAT 검사 모두 Float32Array에 직접 읽기/쓰기, stack 변수만 사용
  • 모듈 레벨 상수: ZONE_KEYS, SENSOR_QUAD_IDX, BODY_QUAD_IDX 등 반복 사용 배열을 모듈 스코프에 미리 할당
  • tempRange 재사용: getProjectedRange()에서 { min, max } 객체를 매번 생성하지 않고 모듈 레벨 싱글톤 재사용

충돌 체크 주기 (collisionCheckInterval)

CollisionConfig.collisionCheckInterval (기본값 33ms)을 통해 차량별로 충돌 검사 주기를 제어합니다. 매 프레임이 아니라 일정 간격으로만 검사하여 CPU 부하를 줄이며, 검사를 스킵하는 프레임에서는 이전 HIT_ZONE 값이 유지됩니다.

EdgeVehicleQueue 활용

Edge 내 차량 목록을 EdgeVehicleQueue로 관리하여, 충돌 검사 시 전체 차량을 순회하지 않고 같은 Edge의 인접 차량만 검사합니다. Edge당 평균 5~50대 수준이므로 사실상 O(1)에 가까운 성능을 달성합니다.

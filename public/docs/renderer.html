Three.js RendererReact Three Fiber + InstancedMesh 기반 대규모 실시간 렌더링

1. 개요

  • React Three Fiber + Three.js InstancedMesh — 선언적 React 구조 위에서 GPU-instanced 렌더링
  • 100k+ 차량 실시간 렌더링 — draw call 1회로 전체 fleet 렌더
  • SharedArrayBuffer → GPU 직접 전달 — Worker가 쓴 위치 데이터를 Main Thread가 읽어 GPU로 업로드

스레드 분리 원칙

VPS는 시뮬레이션 계산(Worker Thread)과 렌더링(Main Thread)을 완전히 분리합니다. Main Thread는 물리 계산, 충돌, 경로탐색을 일절 하지 않으며, Worker Thread는 DOM이나 Three.js에 접근하지 않습니다.

Worker Thread (SimulationEngine) 물리 계산, 충돌 검출, 경로 탐색, Lock 관리 매 프레임 차량 위치를 계산하여 SharedArrayBuffer에 기록 | | SharedArrayBuffer (zero-copy) | Float32Array views — postMessage 없음, 복사 없음 | v Main Thread (React + Three.js) useFrame(60fps) 루프에서 SharedArrayBuffer를 직접 읽음 | | InstancedBufferAttribute (vec4: x,y,z,rotation) | dataArr.set(renderBuffer) — render 전용 버퍼에 복사 | v GPU (WebGL) InstancedMesh가 vec4 attribute를 vertex shader에서 읽어 100k+ 차량을 draw call 1회로 렌더링

핵심 데이터 흐름

  1. Worker가 매 프레임 차량의 (x, y, z, rotation)을 계산하여 SharedArrayBuffer에 직접 write
  2. Main Thread의 useFrame이 SharedArrayBuffer를 read → GPU 업로드용 InstancedBufferAttribute에 .set()으로 복사
  3. GPU가 vertex shader에서 vec4를 읽어 position + rotation 적용 후 렌더링

postMessage는 사용하지 않습니다. 양쪽 스레드가 같은 메모리를 직접 참조합니다 (Worker=WRITE, Main=READ ONLY). 전체 파이프라인에서 JS 객체 할당은 0건(Zero-GC)입니다.

Buffer 설정

const attr = new THREE.InstancedBufferAttribute(float32Array, 4);
attr.setUsage(THREE.DynamicDrawUsage);  // per-frame 갱신 힌트
instancedMesh.geometry.setAttribute('instanceData', attr);

useFrame 루프

useFrame(() => {
  // 1. SharedArrayBuffer(render buffer)에서 Float32Array 읽기
  const src = renderBufferView;  // Worker가 매 step 기록

  // 2. InstancedBufferAttribute로 복사
  attr.array.set(src);

  // 3. GPU 업로드 플래그
  attr.needsUpdate = true;
});

3. Vertex Shader

표준 modelViewMatrix 대신 custom vertex shader에서 instance 변환을 직접 수행합니다.

// vertex shader (GLSL)
attribute vec4 instanceData;   // (x, y, z, rotation_deg)
uniform float uVehicleScale;   // LOD scale factor

void main() {
  // deg → rad 변환
  float rot = instanceData.w * 0.017453292519943295;

  // Z축 회전 행렬 구성
  float c = cos(rot);
  float s = sin(rot);
  mat3 rotateZ = mat3(
     c, -s, 0.0,
     s,  c, 0.0,
    0.0, 0.0, 1.0
  );

  // 최종 위치 = 회전 * (position * LOD_scale) + offset
  vec3 transformed = rotateZ * (position * uVehicleScale)
                   + instanceData.xyz;

  gl_Position = projectionMatrix * modelViewMatrix
              * vec4(transformed, 1.0);
}

deg → rad 상수: 0.017453292519943295 = PI / 180. Worker가 degree 단위로 기록하므로 shader에서 변환합니다.

5. LOD (Level of Detail)

Camera Z 기반 동적 스케일링 — 시각적 효과만, 물리/센서에 영향 없음

// LOD 계산 (Main Thread, useFrame)
const camZ = camera.position.z;
const lod = Math.max(1.0, Math.min(3.5, 1.0 + (camZ - 50) / 200));

// Uniform으로 shader에 전달
material.uniforms.uVehicleScale.value = lod;

6. 선택 차량 Glow Effect

사용자가 차량을 클릭하면 15-layer additive blending glow가 표시됩니다.

Glow 구성

ParameterValueDescription
Layers15겹쳐서 렌더링하는 반투명 레이어 수
BlendingAdditiveBlending빛이 합산되는 효과
Opacity curve0.5 * (1 - t)^2Quadratic fade (안쪽 밝고 바깥쪽 어두움)
Color gradientCyan → Purple#4ecdc4 → #8b5cf6

// Glow layer 생성 (conceptual)
for (let i = 0; i < 15; i++) {
  const t = i / 14;                           // 0 ~ 1
  const opacity = 0.5 * Math.pow(1 - t, 2);  // quadratic fade
  const color = cyan.clone().lerp(purple, t); // gradient
  const spread = lerp(nearSpread, farSpread, distanceFactor);
  const scale = 1.0 + t * spread;

  // additive blending mesh at scale
  renderGlowLayer(position, scale, color, opacity);
}

Distance-adaptive spread: 카메라가 가까우면 glow가 작고 선명하게(0.12), 멀면 크고 부드럽게(2.5) 퍼집니다. 어떤 줌 레벨에서도 자연스러운 시각 피드백을 제공합니다.

7. Worker → GPU 데이터 흐름

Worker Thread SimulationEngine.step() ├─ 차량 위치/회전 계산 └─ Float32Array.set() → SharedArrayBuffer (render buffer) │ │ zero-copy (같은 메모리 주소) ↓ Main Thread useFrame() 루프 ├─ renderBufferView 읽기 (Float32Array view) ├─ InstancedBufferAttribute.array.set(src) └─ attr.needsUpdate = true │ ↓ GPU Upload WebGL bufferSubData() ├─ attribute buffer → GPU memory └─ vertex shader에서 instanceData로 접근 │ ↓ WebGL Render drawArraysInstanced() └─ 1 draw call로 100k+ 차량 렌더링

핵심: Worker → SharedArrayBuffer → Main Thread useFrame → InstancedBufferAttribute → GPU Upload → WebGL Render. 전체 경로에서 JS 객체 할당 0건, postMessage 0회.

9. 최적화 기법

Zero-GC (Garbage Collection 회피)

useMemo로 Matrix4, Vector3, Quaternion, Color 등을 컴포넌트 마운트 시 한 번만 생성하고 매 프레임 재사용합니다. useFrame 내에서 new 호출 0건.

// 마운트 시 1회 생성
const tmpMatrix = useMemo(() => new THREE.Matrix4(), []);
const tmpVec    = useMemo(() => new THREE.Vector3(), []);
const tmpQuat   = useMemo(() => new THREE.Quaternion(), []);
const tmpColor  = useMemo(() => new THREE.Color(), []);

useFrame(() => {
  // tmpMatrix, tmpVec 등을 재사용 — GC 트리거 없음
});

Batch Update

전체 instance 갱신 루프를 완료한 후 한 번만 needsUpdate = true를 설정합니다. 중간에 설정하면 불필요한 GPU 업로드가 발생할 수 있습니다.

Frustum Culling 비활성화

frustumCulled = false — 100k+ instance에서 CPU-side frustum check 오버헤드가 GPU 렌더링 비용보다 큽니다. 전체를 항상 렌더링하는 것이 더 빠릅니다.

4x 메모리 절약. 회전이 Z축 단일이므로 full matrix 불필요. Shader에서 deg → rotateZ 행렬을 직접 구성합니다.

2-Phase Vehicle Text 최적화

100k+ 차량의 8자 라벨(VEH00000)을 매 프레임 갱신하는 핵심 기법입니다.

Phase 1: Vehicle-Level LOD (경량 체크)

  • 전체 차량 순회 — 카메라 거리만 체크 (LOD_DIST_SQ = 20²)
  • Early exit: 축별 거리 먼저 비교 (full distance calc 회피)
  • Uint8Array로 hidden 상태 추적 — 이미 숨긴 차량 skip
  • 가시 차량만 compact Int32Array에 수집 (100k → ~1k)

Phase 2: Character-Level Update (가시 차량만)

  • ~1k 차량 × 8 chars = ~8k 문자만 transform 갱신
  • Billboard quaternion 계산 (카메라 방향으로 회전)
  • dirtyMeshSet: 변경된 InstancedMesh만 needsUpdate = true
100,000 vehicles × 8 chars = 800,000 characters | | Phase 1: Vehicle LOD (distance check) v ~1,000 visible vehicles (99% culled) | | Phase 2: Character transform v ~8,000 character matrices updated | | Selective GPU upload (dirty meshes only) v 39 InstancedMesh.needsUpdate (변경된 것만)

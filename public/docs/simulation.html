Simulation EngineSimulationEngine, FabContext, 시뮬레이션 루프

Overview

시뮬레이션 엔진은 Worker 스레드에서 실행되며, 각 Fab(공장)별로 독립된 FabContext가 물리, 충돌, Lock, 라우팅을 처리합니다.

Simulation Step Pipeline

매 프레임(60fps) 실행되는 5단계 파이프라인:

Step 1: Collision Check └─ SAT 알고리즘으로 센서 영역 충돌 검출 hitZone: -1(없음) | 0(접근) | 1(감속) | 2(정지) Step 2: Lock Management └─ 합류점(merge node) 접근 제어 FIFO or BATCH 전략으로 Lock 승인/해제 Step 3: Movement Update └─ 물리 계산 → 엣지 전환 → 위치 보간 속도/가속도 → ratio 계산 → x,y,z 좌표 Step 4: Auto Routing └─ 경로 재탐색 (Dijkstra) rerouteInterval에 따라 주기적 또는 도착 시 Step 5: Render Buffer Write └─ 차량 위치를 Render Buffer에 복사 (Main Thread 읽기용)

Lock Management

Merge Node Lock

여러 엣지가 합류하는 노드에서 한 번에 하나의 차량만 진입 허용.

  • Request Distance: 합류점 5.1m 전에 Lock 요청
  • Wait Distance: 합류점 2.9m 전에 대기
  • Grant Strategy: FIFO (선착순) 또는 BATCH
  • Auto Release: 차량이 합류점을 지나면 자동 해제

Movement Physics

4-Phase Pipeline

  1. calculateVehiclePhysics — hitZone에 따라 목표 속도 결정, 가속도 계산, 새 속도/ratio 산출
  2. processVehicleTransition — ratio > 1.0이면 다음 엣지로 전환, EdgeVehicleQueue 업데이트
  3. updateVehiclePosition — 엣지 위 ratio → 월드 좌표(x,y,z) 보간, 센서 포인트 업데이트
  4. commitVehicleState — 계산된 값을 vehicleDataArray에 기록

Performance Optimization

Zero-GC Design

  • Dijkstra MinHeap: 단일 number[]에 인덱스 연산 (오브젝트 생성 없음)
  • Physics scratchpad: SCRATCH_PHYSICS 사전 할당 재사용
  • Collision: SAT O(1), 메모리 할당 없음
  • Typed Array만 사용 (hot path에서 오브젝트 할당 금지)

Performance Metrics (5초마다 리포트)

  • avgStepMs, minStepMs, maxStepMs, p50/p95/p99
  • Dijkstra count/avgTime
  • Fab별 차량 수

# 실시간 통계 차트 시각화 — 개발 계획

> **목표**: 24 fab 평행우주 routing/lock ablation 결과를 한눈에 비교 가능한 dashboard. **Master-Detail + Tab 기반** UX로 개선.
>
> **현 상태**: `FabStatsPanel.tsx` (682 lines) — Cards / Compare / Trend 3-tab 구조 이미 존재. 이번 작업은 **greenfield가 아니라 redesign**.

---

## 0. TL;DR — 내 의견 요약

1. **이미 있는 걸 다 갈아엎지 말고 단계적으로 마이그레이션** 하자. 현재 `Cards` 탭은 사실상 "랭킹 탭의 Master + 그리드형 카드"의 변형이고, `Trend` 탭은 시간축 차트 기능이 살아있다. 재활용 가능한 자산이 많다.
2. **데이터 갭이 진짜 리스크**다. UI 레이아웃보다 데이터 소스가 핵심 병목:
   - **Histogram (Lead time 분포)**: 현재 `OrderStatsData`에 p50/p95/mean만 있고 **raw samples 없음** → Worker에서 histogram bucket을 보내거나, reservoir sampling을 추가해야 함.
   - **Edge 혼잡도 Top 10**: `EdgeStatsTracker`는 존재하지만 main thread로 노출되지 않음. 정기적으로 Top N edge를 postMessage해야 함.
   - **Lock wait avg KPI**: `OrderStatsData`에 없음. Lock 시스템에서 별도로 집계해서 export 필요.
   - **History ring buffer**: 현재 `useRef Map`에 휘발성으로 저장 → panel 닫으면 사라짐. **Zustand로 옮겨야** Tab 전환/패널 재오픈 시 데이터 유지 가능.
3. **Box Plot 때문에 ECharts를 추가하는 건 비추**. Bundle 사이즈 부담 큼 (~700KB). 3~5개 그룹밖에 안 되니 **SVG로 직접 그리는 BoxPlot 컴포넌트 100줄**이면 충분. Recharts ErrorBar 변형으로도 가능. 정 필요하면 마지막 단계에서 평가.
4. **현 `FabStatsPanel.tsx` 682줄을 절대 한 파일에 더 욱여넣지 말 것**. 폴더 구조로 쪼개야 함 (`panels/FabStats/`).
5. **Phase 0 (선행 작업)**을 명시적으로 추가하자: 데이터 백엔드 정비. 이게 안 되면 Phase 2부터 막힘.

## 2. 데이터 백엔드 보강 (Phase 0)

> **모든 UI 작업의 전제**. 이걸 먼저 안 하면 Phase 2~3에서 막힌다.

### 2-1. `OrderStatsData` 확장

Worker → Main으로 보내는 payload에 추가:

```typescript
interface OrderStatsData {
  simulationTime: number;
  completed: number;
  throughputPerHour: number;
  leadTimeP50: number;
  leadTimeP95: number;
  leadTimeMean: number;
  totalPathChanges: number;

  // ── 신규 ──
  /** 0-10s, 10-20s, ..., 590-600s, 600+ 등 fixed bucket. histogram chart용 */
  leadTimeHistogram?: number[];        // length 13 (12 bucket + overflow)
  leadTimeBucketSizeSec?: number;      // 예: 10

  /** Lock 시스템 평균 대기 시간 (ms 또는 sec). KPI 카드용 */
  lockWaitAvgMs?: number;
  lockWaitP95Ms?: number;

  /** Top N congested edges (혼잡도 = 대기 vehicle 수 또는 평균 wait time) */
  topCongestedEdges?: Array<{ edgeName: string; metric: number }>;  // length 10
}
```

**책임 위치**:
- Worker 측: `shmSimulator/managers/OrderStatsCollector` (또는 동등 위치)에서 raw lead time 들어올 때마다 bucket 카운트 증가. flush 주기마다 함께 보냄.
- Lock wait: Lock 시스템에서 grant timestamp - request timestamp 집계. 기존 logger 데이터 활용 가능한지 확인 (lock_redesign_v04.md 참고).
- Edge congestion: `EdgeStatsTracker` 매 N step마다 `getTopN(10)` 호출해서 collector에 push.

### 2-2. `orderStatsStore` 확장: 시계열 ring buffer를 Zustand로

현재 `useRef`로만 들고 있어서 휘발성 → Zustand에 ring buffer 영구화:

```typescript
interface FabTimeSeriesEntry {
  t: number;                  // sim seconds
  throughputPerHour: number;
  leadTimeP50: number;
  leadTimeP95: number;
  vehicleStates: { idle: number; moveToLoad: number; loading: number; moveToUnload: number; unloading: number };
  collisionCount: number;
  lockWaitAvgMs: number;
}

interface OrderStatsStore {
  fabStats: Record<string, FabOrderStats>;              // 기존
  fabHistory: Record<string, FabTimeSeriesEntry[]>;     // 신규, 길이 cap N (예: 600)
  groupHistory: Record<string, GroupAggEntry[]>;        // 신규, 그룹 평균 시계열

  pushFabHistory: (fabId: string, entry: FabTimeSeriesEntry) => void;
  // ... 기타
}
```

**Cap**: 600 entries × ~12 numbers = ~30KB/fab × 24 fabs = ~700KB. 무난.

### 2-3. Update 주기

- **현재**: `UPDATE_INTERVAL_MS = 2500ms` (2.5초). main thread setInterval.
- **유지**: 이 주기로 시계열 push. 더 자주는 비추 (re-render 폭증).
- 차트 컴포넌트는 `useOrderStatsStore` selector로 history slice 구독 → React가 알아서 diff.

### 3-2. Zustand store 분리

기존 `orderStatsStore`는 **데이터 저장소**로 두고, UI 상태는 별도 store로 분리:

```typescript
// FabStats/store.ts
interface FabStatsUIStore {
  activeTab: 'ranking' | 'group';
  rankingSortBy: MetricKey;
  rankingSortOrder: 'asc' | 'desc';
  rankingFilterStrategy: 'all' | 'DISTANCE' | 'BPR' | 'EWMA';
```

**이유**: 데이터 store와 UI store를 섞으면 `selectedFabId` 변경 시에도 모든 history 구독자가 re-render됨. 명확히 분리.

## 5. 리스크 및 의사결정 포인트

### 🔴 R1: Lead time histogram 데이터 소스

**문제**: Worker 측에서 lead time을 어디서 어떻게 수집하는지 미확인. 만약 raw sample이 일정 cap으로만 유지되고 있다면 histogram을 만들 raw data가 부족할 수 있음.

**결정 필요**:
- (A) Worker에서 bucket counter 누적 (작업 추가)
- (B) Reservoir sampling으로 raw 1000개 유지 후 main thread에서 bucketize
- **권장: (A)**. 데이터 양 일정하고 계산 적음.

### 🔴 R2: Lock wait avg KPI의 정의

- Lock 시스템 v0.4 메모리에 따르면 zone preemption 제거 + 단순 FIFO. wait time = REQUEST → GRANT 간격.
- 이게 이미 logger에 기록되고 있는데, **실시간으로 main thread에 노출하려면 별도 채널** 필요. 기존 OPFS flush는 너무 느림.
- **결정**: OrderStatsCollector에 hook 추가해서 매 flush마다 최근 N개 평균 함께 보냄.

### 🟡 R3: Box Plot 라이브러리 선택

- ECharts: ~700KB, 풀 feature. 다른 곳에서도 쓸 거면 OK.
- Recharts ErrorBar: 워크어라운드. 미려하진 않음.
- **SVG 직접 그리기**: 100줄, 풀 컨트롤, 번들 부담 0.
- **권장: SVG 직접 그리기**. 그룹 수 3~5개 고정이고, 디자인 톤 통일성도 좋음.

### 🟡 R4: 실시간 vs 스냅샷

- 24 fab × 매 2.5초 update × 차트 4개 × ResponsiveContainer 측정 → 측정 안 한 채로는 알 수 없는 성능 비용.
- **완화책**: 
  - `isAnimationActive={false}` 모든 차트 (이미 적용됨)
  - Detail은 선택된 fab만 렌더 (이미 그 구조)
  - Master는 row 24개라 부담 없음
  - `useShallow` 또는 selector 좁히기로 re-render 최소화
- **검증**: Phase 2 끝나면 React Profiler로 한 번 측정.

### 🟢 R5: 기존 사용자 경험 깨짐 방지

- 누군가 `cards` / `compare` / `trend` 탭 단축어를 외워서 쓰고 있을 수 있음 → 큰 우려는 아니나 Phase 1에서 임시 폴백 두면 안전.
- 더 큰 우려: Reset KPI 버튼 (L630~639) 동작 유지 필수.

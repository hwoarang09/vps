Logging & Data PipelineOPFS 바이너리 로그, MQTT, Python 분석

Architecture

SimulationEngine.step() ├─ lockMgr → onLockEvent → logLock() ├─ movement → onEdgeTransit → logEdgeTransit() ├─ autoMgr → onPathFound → logRoute() ├─ orderMgr → complete → logOrderComplete() └─ every frame → isReplay? → logReplaySnapshot() │ ↓ SimLogger.logXXX() │ ↓ EventBuffer (ArrayBuffer, count/view/handle) │ ↓ count ≥ 512 OR 10s periodic flush() { OPFS: handle.write(slice, {at: offset}) ← sync write MQTT: DbShipper.push() ← optional }

Why OPFS?

  • Worker에서 동기 쓰기 — IndexedDB/FS Access API는 비동기. OPFS는 createSyncAccessHandle()로 즉시 쓰기.
  • 대용량 — localStorage 5MB 한계 없음. GB 단위 가능.
  • 고속 바이트 쓰기 — 직렬화/트랜잭션 오버헤드 없음.
  • 권한 불필요 — 브라우저 격리 파일시스템, 사용자 프롬프트 없음.

Binary Format

Little-endian, 고정 크기 레코드. DataView로 직접 쓰기. JSON/직렬화 없음.

// 예: ML_EDGE_TRANSIT (24 bytes)
view.setFloat32(offset + 0, ts, true);       // timestamp
view.setUint16(offset + 4, vehId, true);     // vehicle ID
view.setUint16(offset + 6, edgeId, true);    // edge ID
view.setFloat32(offset + 8, enterTs, true);  // enter time
view.setFloat32(offset + 12, exitTs, true);  // exit time
view.setFloat32(offset + 16, edgeLen, true); // edge length
// padding to 24B

SnapshotLogger

가변 크기 블록 포맷으로 전체 차량 상태 스냅샷 저장 (10초마다).

[MAGIC=0xCAFE (2B)][ts (4B)][numVehicles (2B)]
  [vehId (2B)][edge (2B)][ratio (f4)][velocity (f4)][stopReason (2B)] × N
[numActiveEdges (2B)]
  [edgeId (2B)][vehCount (2B)][vehId (2B) × count] × M

File Naming

OPFS 파일 경로:
  {sessionId}_{fabId}_{eventType}.bin     ← 고정 크기 이벤트
  {sessionId}_{fabId}_snapshot.bin        ← 가변 크기 스냅샷

예: 1716100800000_fab_0_0_edgeTransit.bin

Python Analysis Pipeline

OPFS .bin 파일 → 브라우저에서 다운로드 ↓ log_parser.py ├─ iter_file(path, suffix) — streaming 파서 (O(1) 메모리) └─ 고정 크기 레코드를 yield ↓ analyze.py ├─ cmd_summary — 파일별 통계 ├─ cmd_lock_node — 노드별 Lock 분석 └─ cmd_checkpoint — 체크포인트 분석
